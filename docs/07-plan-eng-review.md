# Engineering Review: 等我有钱了

## 工程判定

当前不应继续投资完整移动端工程。

在产品形态未确定前，App 工程会制造假进度：看起来在开发，实际上没有验证用户为什么要用。

## 当前原型状态

当前仓库里有一个 Expo spike：

- `App.tsx`
- `app.json`
- `package.json`
- `package-lock.json`
- `tsconfig.json`

它可以证明 React Native/Expo 能跑，但不证明产品成立。

后续处理方式：

- 暂不继续开发。
- 不把它作为主线架构。
- 等内容验证通过后，决定保留、重写或删除。

## 推荐技术路径

### 阶段 0: 文档与内容验证

只维护 Markdown 文档和内容样例。

输出：

- 定位
- 风险边界
- 内容样例
- 用户反馈
- 下一步决策

### 阶段 1: H5 生成器

如果内容验证通过，先做 Web/H5，而不是 App。

原因：

- 分享链路更短。
- 不需要应用商店审核。
- 可以快速投放到微信群、小红书、朋友圈。
- 更适合验证“输入 -> 生成 -> 分享”闭环。

推荐栈：

- React + Vite 或 Next.js
- 本地模板生成，先不接 AI
- 生成图片可后置

### 阶段 2: App

只有在 H5 证明传播和留存后，再考虑 App。

App 适合承载：

- 愿望清单
- 每日内容
- 用户历史
- 分享素材库
- 订阅或高级模板

## 工程门禁

进入实现前必须具备：

- 明确选择 A/B/C 哪个内容方向。
- 至少 10 条高分样例。
- 第一版页面流程图。
- 上架/传播风险检查。
- 验收标准。

## 最小实现建议

如果内容验证通过，第一版只做一个页面：

1. 输入框：等我有钱了，我要...
2. 生成按钮
3. 三个结果风格：爽文、反差、治愈
4. 复制文案
5. 生成分享卡

不做账号、不做后端、不做支付、不做复杂资产系统。

## 测试策略

Web/H5 阶段：

- TypeScript 检查
- 基础组件测试
- 浏览器截图验证
- 移动端 390px 宽度验证

App 阶段：

- Expo/React Native 类型检查
- 真机截图
- 分享功能测试
- App Store 文案风险检查

## 工程结论

当前主线不是继续写代码，而是完成内容验证。

下一次允许动手写代码的条件：

`docs/05-content-validation-plan.md` 的通过标准达到，且用户明确选择 H5 或 App。
