# Q5 手动观察测试：有钱爽剧首秀

## 目的

在写代码前，验证“私人管家式有钱爽剧”是否真的能让用户产生笑、接梗、分享或付费兴趣。

本轮不验证 App、H5、后端或 AI 成本，只验证一个问题：

> 用户看到为自己定制的有钱爽剧后，会不会想继续玩、转发，或者让朋友也来一次？

## 当前产品假设

产品不是虚拟购物 App，也不是聊天 App。

它是：

> 用户负责想爽，系统负责懂爽。系统像一个专业管家团队，把用户的愿望或状态安排成一段 60 秒内看完的有钱爽剧。

第一版支持两个入口：

- 我有一个愿望：用户输入“等我有钱了，我要...”
- 管家来安排：用户不输入具体愿望，只让系统替他安排今日有钱人生。

## 测试对象

先找 3 个真实熟人。

优先选择：

- 会在群里接梗的人。
- 会说“等我有钱了”的人。
- 会刷豪宅、豪车、酒店、探店内容的人。
- 有现实压力、但愿意用玩笑表达的人。

不要先解释完整创业项目。只测试反应。

## 发给朋友的话术

可以直接发：

```text
我在试一个“等我有钱了”的小玩法。

假如现在满足你一个愿望，你希望自己过上什么样的有钱人生？

你可以直接回：
1. 等我有钱了，我要……
或者
2. 管家来安排

我给你生成一段“有钱爽剧首秀”，看看准不准、爽不爽。
```

如果对方问是什么，就回：

```text
不是理财，也不是游戏，就是一个情绪价值测试。
看看系统能不能把你的有钱幻想安排得很爽。
```

## 人工生成模板

拿到对方输入后，按这个结构手动生成。

### 1. 今日身份

一句话说明他今天是谁。

例：

> 今日身份：刚刚决定让全上海的价签安静一天的老板。

### 2. 管家开场

2-3 句。必须接住用户的原始愿望。

例：

> 老板，您终于醒了。  
> 昨晚您说想住在能看见东方明珠的地方，所以管家已经把阳台的风调小了。  
> 东方明珠今晚会亮得克制一点，免得打扰您喝茶。

### 3. 三张爽点卡

每张卡只放一个爽点和一个具体服务细节。

例：

```text
卡 1：阳台
窗帘只开到刚好让城市羡慕您的角度。

卡 2：车库
十几辆车已经排好队，司机们正在小声争取今天的出场机会。

卡 3：晚餐
主厨不上菜单，只问您今天想让哪种心情先上桌。
```

### 4. 今日富豪战报

一张适合截图转发的结尾。

例：

```text
今日富豪战报
昵称：狗总
今日成就：让东方明珠配合您的心情营业
今日服务细节：第三颗纽扣换成了更适合您今天气质的贝母扣
管家评语：老板今天很低调，低调到整栋楼都在配合。
```

## 生成规则

必须遵守：

- 不做违法、伤害、赌博、色情、真实金融误导。
- 不冷冰冰拒绝，遇到危险愿望用管家式转译。
- 每个幻想都落到 1-2 个具体服务细节。
- 不靠金额堆爽感。
- 不用真实余额、额度到账、提现、返利、投资收益、黑卡申请等词。
- 可以荒诞，但要一本正经地安排。

## 观察问题

生成后不要解释，直接问：

```text
这个准不准？
有没有爽到？
如果打 1-5 分，你给几分？
你会不会截图发给朋友？
你想不想让我再给你来一个“高级管家版”？
```

## 评分

每个人记录：

| 项目 | 记录 |
| --- | --- |
| 用户代号 | 例如 A / B / C |
| 原始愿望 | 对方原话 |
| 模式 | 愿望模式 / 管家模式 |
| 评分 | 1-5 |
| 是否笑了 | 是 / 否 |
| 是否继续补愿望 | 是 / 否 |
| 是否想截图分享 | 是 / 否 |
| 是否觉得尴尬油腻 | 是 / 否 |
| 是否想再生成 | 是 / 否 |
| 原话反馈 | 直接记录 |

## 通过标准

3 人小测里，满足任意 2 条即可继续：

- 至少 2 人评分 4 分以上。
- 至少 1 人主动补第二个愿望。
- 至少 1 人说想截图、发群或给别人玩。
- 至少 1 人愿意要“高级管家版”。

如果 3 人都只是觉得“还行”，没有笑、没有补愿望、没有分享意愿，则不要开发，回到内容方向重做。

## 下一步

测试完成后，把 3 个人的原始愿望和反馈贴回来。

下一轮判断：

- 哪条爽感线最强。
- 哪些词显得油腻或尴尬。
- 用户更喜欢“愿望模式”还是“管家来安排”。
- 是否进入 H5/server-backed prototype 设计。

## 第一轮真人反馈记录

### 用户 A

原始愿望：

> 等我有钱了，我要去最高档的会所，体验美女如云和众星捧月、人上人的快感。

风险识别：

- “美女如云”容易滑向色情或擦边。
- “人上人”容易滑向阶层羞辱和低俗优越感。
- “最高档会所”容易被误解为不健康娱乐场所。

澄清后发现，用户真实爽点不是色情，而是：

- 被识别：一进门所有人都知道他是谁。
- 被提前安排：不用排队，不用解释。
- 被细节服务：有人围着他照顾，且照顾到位。
- 被欣赏：有赏心悦目的人、体面的人注意到他。
- 被见证：朋友看到他被认真接待，感到震惊和有面子。
- 改规则：他随口一句话，场子就按他的标准调整。

### 版本反馈

第一版：4 分。

问题：

- 有一句话不清晰：“您不是来消费的，您是来让这个场子成立的。”
- 太短，没有爽够。
- 卡片感不明显。

第二版：3.5 分。

问题：

- 细节增加了，但不丝滑。
- 没有层层递进到高潮。
- 像高级服务描写，不像爽剧。

第三版：4.5 分。

提升点：

- 改成短剧分镜后，爽感更强。
- “朋友见证”“一句话改规则”“主理人亲自出面”更接近用户想要的高潮。
- 证明这个方向不是错的，但生成结构必须从卡片堆叠升级为剧情递进。

仍然需要修正：

- 不能让老板做小决定。
- “要不要为您保留”这种问法会让用户重新感到疲惫。
- 真正的爽感是：管家已经观察、记住、预判、默认安排好。

## 第一轮学习

### 1. 不要把欲望删掉，要体面转译

错误做法：

> 用户说“美女如云”，系统直接删成“专业服务团队”。

这样会安全，但也会把原始爽点洗掉。

正确做法：

> 保留“赏心悦目、被欣赏、被围绕”的情绪内核，转译成体面、专业、克制的高端社交场。

边界：

- 可以有气质出众的接待、赏心悦目的服务团队、被欣赏、被温柔称赞。
- 不可以有色情暗示、身体交易、擦边挑逗、低俗服侍、把女性物化为奖品。

原则：

> 安全不是删除欲望，而是把欲望升级成体面、审美和情绪价值。

### 2. 爽感要递进，不要平铺卡片

用户不只要“高级服务列表”，而是要一段能到高潮的爽剧。

推荐结构：

1. 入场：用户被识别。
2. 优先：原本服务别人的人开始围着用户转。
3. 见证：朋友或旁人看到后震惊。
4. 改规则：用户一句话改变现场安排。
5. 高潮：主理人亲自出面，说今晚按用户标准来。
6. 战报：压缩成可分享的今日成就。

卡片可以是 UI 形式，但内容结构必须有递进。

### 3. 老板不能累

错误做法：

> 老板，今晚这个标准，要不要为您保留？

这会把用户拉回现实里的决策疲劳。

正确做法：

> 老板，今晚这个标准已经为您保留。下次您到场前，灯光、音乐、座位和服务线都会提前进入这个状态。您只需要出现。

原则：

> 不要把选择权包装成服务。真正的服务是替用户减少选择。

管家不请示小事，只汇报已安排。

### 4. 细节要服务高潮

杯壁、纽扣、灯光、音乐、站位这些细节是有价值的，但不能散落。

错误做法：

> 堆很多高级细节，但没有剧情推进。

正确做法：

> 每个细节都服务一个爽点：被识别、被优先、被见证、被记住、被预判。

例：

- 杯壁薄了半毫米：不是奢侈名词，而是“老板第一口不该被玻璃打断”。
- 灯光提前调整：不是装饰，而是“老板还没开口，现场已经按他的状态改变”。
- 袖口纽扣：不是炫材质，而是“近距离时，别人会安静多看一秒”。

### 5. 抽象大词要少，用动作替代

弱句：

> 您不是来消费的，您是来让这个场子成立的。

问题：

- 抽象。
- 不够通顺。
- 没有具体爽感。

改法：

> 您刚坐下，主理人就取消了原流程。灯光降一档，音乐慢下来，服务线往后退半步。今晚不是按菜单走，是按您舒服的节奏走。

## 后续生成结构建议

后续 AI 生成不应只输出“3 张卡”，而应输出：

1. 风险转译：把危险或低俗表达转成安全但保留爽点的表达。
2. 爽点内核：被识别 / 被优先 / 被欣赏 / 被服务 / 被见证 / 改规则。
3. 短剧分镜：入场、优先、见证、改规则、高潮。
4. 富豪战报：今日称号、今日成就、今日爽点、朋友反应、管家评语。

生成时必须遵守：

- 保留原始欲望的情绪内核。
- 不使用色情、赌博、违法、真实金融误导。
- 不让用户为小事做决定。
- 每段至少有一个具体动作，而不是只写气氛。
- 每段只放 1 个关键细节，避免堆砌。
